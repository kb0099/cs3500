PS3 - Formula Class

Problem Set 3 - A Formula Class

For this assignment you will be creating a Formula class which is a more generalized version of your FormulaEvaluator work. This formula class will be utilized in the upcoming spreadsheet assignment. In PS1, the evaluate function stood alone: it was only used once, returned a result, and then it's "life was over". In a spreadsheet the formula in a given cell will be used over and over again, as many times as the dependent cells are modified. Thus a Formula object must store the expression to evaluate (e.g., "1+2") along with any other needed information (e.g., dependencies), and can then be used to "re-evaluate" the expression whenever necessary (e.g., when other cells get changed). (Note: the result of this assignment is still not a spreadsheet, i.e., your formula class is not responsible for keeping track of modifications to other cells, that will be a future assignment.)

Implementation

Your starting point will be the PS3Skeleton solution that you will find in the class repository. It contains a partially implemented project called Formula. It is a generalization of the infix expression evaluator that you created for PS 1. You are to create a solution called PS3 and complete the implementation of the Formula project. You are not allowed to modify any of its public declarations, but can, of course, add private data and helper methods.

The incomplete portions are all in the Formula class. You should be able to reuse much of your code from PS1, although some changes will be necessary. This is a good chance to correct any mistakes made on PS1.

Please be aware that the specification has changed slightly and you need to be careful to catch all the changes. For example: spaces now delimit tokens, meaning x 23 is not the same as the variable "x23".

Immutable

You should make sure that your formula object is immutable. In other words, there is no way to change a formula after it has been created. For example (in future spreadsheet), the only way to modify a cell with a formula in it is to build a new formula (this formula can of course be built by modifying the string representation of the previous formula).

Question: Does this mean the formula always evaluates to the same value?

Question: Does this mean that once you build the spreadsheet part of the program, you cannot change the formula in a cell?

Formula Format Exception

Whenever a formula has incorrect syntax, you are to throw a FormulaFormatException. This class is provided for you. Notice that it is essentially a wrapper for the Base class Exception.

For each error condition, you should provide a meaningful warning message explaining 1) what went wrong and 2) providing a suggested possible fix.

Note: The try/catch exception handling works based on the exception type (i.e., FormulaFormatException), not the exception message. The exception message would eventually be propagated through the system to the user to help them fix the problem.

Floating Point Numbers

For this assignment, we will be converting our integer math into floating point math. This should not require major changes in your code, but as always, think through the implications (e.g., how will you handle E notation 5e-5? how will you handle precision: 0.0000000000000000000001).

Equality

Two Formulas are considered equal if they contain the same tokens in the same order. Remember that white space is ignored. Thus, make sure your code can determine that 2.0 == 2.000 or that X + Y is the same as X+Y.

Question: How are you storing your formula and thus how can you compare two formulas to see if they are the same?

Syntax and Syntax Errors

The Formula constructor should throw an exception if the formula being parsed is syntactically incorrect.

Below we discuss parsing and some possible ways to detect syntactic errors. In each case an appropriate exception and message should be created:

  1. Parsing

    We have provided a private method that will convert an input string into tokens. Verify that the only tokens are (, ), +, -, *, /, variables, and floating-point numbers.

    Question: What are some "boundary case" for floating point numbers?

    Question: can you explain why the GetTokens method uses the yield syntax? Related to this: what does a return type of IEnumerable mean?

  2. One Token Rule

    There must be at least one token.

  3. Right Parentheses Rule

    When reading tokens from left to right, at no point should the number of closing parentheses seen so far be greater than the number of opening parentheses seen so far.

  4. Balanced Parentheses Rule

    The total number of opening parentheses must equal the total number of closing parentheses.

  5. Starting Token Rule

    The first token of an expression must be a number, a variable, or an opening parenthesis.

  6. Ending Token Rule

    The last token of an expression must be a number, a variable, or a closing parenthesis.

  7. Parenthesis Following Rule

    Any token that immediately follows an opening parenthesis or an operator must be either a number, a variable, or an opening parenthesis.

  8. Extra Following Rule

    Any token that immediately follows a number, a variable, or a closing parenthesis must be either an operator or a closing parenthesis.

Pre-Evaluation of Formula

There are two stages at which you "use" a Formula object. (1) When you create the initial object, and (2) when you evaluate the current status of the object.

  1. Construction of a Formula object

    At this point, any syntactically incorrect formula should be flagged via the exception process. An "invariant" will then be set saying that any formula object (that survived the original checks) is always a valid formula.

    Note: a valid formula can still result in an invalid result, e.g., X / Y

  2. Evaluation of the current value of the Formula object

    The Evaluate method should use the evaluation algorithm from PS 1, modified to deal with doubles instead of integers and to take account of the normalization delegate.

    Because the constructor throws an exception when it encounters a syntactically incorrect formula, the only problems that the Evaluate method needs to worry about are undefined variables and division by zero.

Delegates

For this assignment the "outside world" (i.e., the code that will be using the formula) can optionally provide two pieces of information at formula construction time (i.e., a normalizer and a validator) and one piece of information at evaluation time (i.e, a lookup function).

The lookup function. The variable lookup function serves the same purpose as before and is provided at Evaluation time.

Normalization

Normalization is a way of putting information into a common form. For example:

In Excel: the cell A1 can be referred to as A1 or a1 (Excel automagically coverts all lower case cell names to upper case). In C# programming: variables can be written as: student_list or StudentList (Visual Studio does not automagically make this transformation, but it could...). In order to allow this flexibility to the user of our eventual spreadsheet, we will allow the creator of a Formula object to provide a function which converts variables to a "canonical" form (this means: to a standard form). One easy "normalizer" function would be a method that lower cases all letters in a variable name

Thus: (1) before/during the syntactic check on the original formula, you should normalize all variable names. (2) The associated lookup function should expect variable names to be passed in using their normalized form. Thus if the expression was A1 + B1, and the normalizer was "lowercase( var )", then the lookup function should store the keys: a1, and b1.

Variable Name Validation

The Validator. The isValid function represents the application's rules for what a valid variable name is. For example, linked_list might be a valid variable in C#, but it would not be a valid variable in Excel. Rather than have the Formula class be the arbitrator of what is a valid variable names (and thus having to define valid variable names for all possible uses), we allow the user of our Formula object to provide a function expressing this information.

Note: if no normalizer and/or validator is provided, then you are to use the identify function ( s → s) for the normalizer and ( s → true ) for the validator. In other words, a1 and A1 would be different variables, and any letter followed by any number of letters, digits, and/or underscores would form valid variable names.

Testing

Part of this assignment is to develop a rich set of tests that can discover a wide range of errors in anyone's (and of course: your) implementation of the formula object. We will be grading your tests as part of the total score for this assignment.

You are to add a test project called FormulaTester to your PS3 solution. It should contain a good set of black-box unit tests that achieve (very close to) 100% code coverage for the source file Formula.cs.

Naming: Name your unit tests with a descriptive title. Additionally: all tests that deal with the public methods, properties, etc., of the class should be placed at the top of your testing file and have a method name that starts with "public" (e.g., public_test_unknown_variable). Any test that is used to test private members of the class should have a name starting with "private" (e.g., private_test_string_rep).

Commenting: Remember, each test should be designed to "break" a specific property of the implementation. You should name and comment your tests to make it explicit what their purpose is. If your test is "tricky" you should especially document what it is attempting to do.

Private Test Placement. Please put all the public tests at the top of your file and all the private tests at the bottom. We will be taking your public unit tests and running them against buggy versions of our completed version of Formula.cs to see if your tests can detect our errors. (Do this we will have to delete your private tests.) Note: If your public tests assume anything else about the classes in Formula.cs, we will not be able to run them and they will not detect any errors.

Question: why are we only going to use your public test cases on our code and not your private test cases.

Some Hints and Thoughts

  • Equality is determined by making sure each token in formula one is the same as that in formula two. You should consider the documentation of the toString function and how this might make your life a lot easier...

  • Some boundary cases for numbers/doubles might be 0, the largest possible number, infinity, NAN, negative numbers, etc.

  • IEnumerable returns a data type that can be iterated over (e.g., using the foreach loop). Yield "saves" the current computation to be "reactivated" as necessary at a future point. Note: when using a foreach loop, the "reactivation" would be at the start of each iteration of the loop.

  • Test cases that check private methods and/or private variables in a class are specific to a given implementation. Test cases that test the results of public methods/contents of public member variables (if any) are useful in evaluating the generic API of the class.

  • Immutable objects (such as strings) are "static" once created. This ensures many nice properties about them, for example if you were to lower case a string upon creation (in the constructor) you would know that the string could never contain capital letters, and thus the toLower method could just return the current string. (Note: I'm not suggesting you would really do this.)

    By making our formula immutable, we can ensure certain properties about it, not the least of is: if it is syntactically correct upon creation it will always be syntactically correct.

    Finally, while we cannot change a formula once it has been "newed", we can always replace the contents of a variable with a new formula.

Rules for grading PS3

Staff Unit Test Cases
These are staff generated grading tests which will attempt to verify that your code meets the required specifications of the assignment. You will be responsible for running these tests yourself and fixing any errors after they are released to the class (after the due date).
Software Practices: Proper Use Versioning/Repository
The purpose of versioning is to allow you to more effectively manage software as it becomes more complex and larger. This includes seeing modifications over time, and being able to return to past instances of the code to see what has changed. To this end, you should use your GitHub account frequently to check in meaningful changes to your code.
Software Practices: Respecting the Job Requirements
The solution respects the requirements of the assignment as defined in the skeleton code, the assignment web page, and during in class discussions. For example: the Formula class is built as a library; there is a good test suite; the namespace and method names are as defined.
Software Practices: Encapsulation, Abstraction, Code Presentation and Complexity Description for Students
Any class you define should be a "self contained", easy to use, easy to maintain, working, functional, implementation of the requirements. The better job you do with documentation, encapsulation, abstraction, etc., the easier it will be for future developers on your code base to successfully navigate and manipulate your code. For example, the member variables that are used to represent a Formula should be declared private. (If properties are used instead, they have private get and set). The code is easy to understand and the use of functions make the code less complex (and easier to test). The choice of data structures is appropriate, and when ever possible built in functionality of these data structures is used.
Software Practices: Student developed Test Cases and Code Coverage Description for Students
Tests are there to help ensure correctness and to show you when new functionality (new code) results in breaking old behavior. Thus there should be a significant number of meaningful test cases which, when they fail, are likely to be informative. Further, the combination of all tests should cause every line of code to be executed (and thus, somewhat evaluated).
Software Practice: Overall (e.g., Transfer to a new developer) Description for Students
The goal of a program is to solve a problem. The goal of a Software Engineer is to build a program that not only solves the current problem, but is also as error free as possible, AND to be maintainable/readable/understandable by the next new person who comes along and is put in charge of using it to solve the next problem. Following requirements, making your code readable, choosing proper data structures, dividing your code into appropriate classes and/or methods, using proper versioning, etc., are all part of good software practice. This category represents the "team leads" overall categorization of your work. Please note that this score is influenced by the above categories and represents additional points associated with those categories as well as the "overall software practice" category.
